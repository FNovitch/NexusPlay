

Use como base o diagnóstico anterior, mas não tente transformar o projeto em uma aplicação pronta para produção comercial. O objetivo é deixá-lo estável, apresentável, seguro no nível necessário para uma demonstração pública e fácil de entender por recrutadores.

Preserve a arquitetura, o design e as funcionalidades já existentes. Não recrie o projeto do zero e não adicione funcionalidades complexas que não sejam necessárias para a apresentação no portfólio.

## Objetivo final

Ao concluir, o projeto deve:

* Possuir front-end, back-end e banco preparados para deploy.
* Abrir sem erros críticos no navegador.
* Ter build do front-end e do back-end funcionando.
* Permitir uma demonstração clara dos principais fluxos.
* Não possuir botões importantes sem funcionamento.
* Não expor credenciais ou dados sensíveis.
* Deixar claro quando uma funcionalidade utiliza dados de demonstração.
* Ter um README profissional.
* Poder ser apresentado honestamente como uma aplicação full stack em ambiente demonstrativo.

## Prioridade 1 — Bloqueadores básicos para deploy

Corrija apenas os problemas realmente necessários para publicar o projeto com segurança.

### Variáveis de ambiente

Revise os arquivos de configuração do front-end e do back-end.

Garanta que:

* `DATABASE_URL` seja usada na conexão com o banco.
* `JWT_SECRET` seja obrigatório em produção.
* `FRONTEND_URL` seja usada no CORS.
* `VITE_API_URL` seja usada pelo front-end.
* Não existam senhas, tokens, chaves ou URLs privadas diretamente no código.
* Os arquivos `.env` estejam ignorados pelo Git.
* Exista um `.env.example` no front-end e no back-end.
* Os exemplos não contenham valores secretos reais.

O front-end pode usar `localhost` como fallback apenas em desenvolvimento. Em build de produção, deve utilizar a variável de ambiente configurada.

### CORS

Revise:

`backend/src/app.ts`

Configure o CORS para:

* Permitir localhost somente em desenvolvimento.
* Permitir somente o domínio do front-end em produção.
* Utilizar `FRONTEND_URL`.
* Não deixar uma configuração aberta para qualquer origem.

### Login e autenticação

Adicione rate limit ao endpoint genérico de login em:

`backend/src/routes/auth.routes.ts`

Use uma implementação semelhante às rotas de login que já possuem limitação.

Garanta também que:

* `JWT_SECRET` não tenha fallback inseguro.
* Tokens e senhas não apareçam nos logs.
* O logout limpe corretamente o estado salvo no front-end.
* Uma sessão inválida não continue permitindo acesso às páginas protegidas.

### Mercado Pago

Como o projeto será usado para portfólio, não é necessário ativar pagamentos reais.

Ajuste o comportamento para que:

* Em modo demonstração, o checkout possa ser simulado ou utilizar sandbox.
* Em produção real, o webhook exija `MERCADO_PAGO_WEBHOOK_SECRET`.
* Caso as credenciais não estejam configuradas, a aplicação exiba uma mensagem clara em vez de quebrar silenciosamente.
* Nenhum pagamento real seja executado durante os testes.

Documente no README que a integração está preparada para sandbox ou demonstração.

### Uploads

Não é obrigatório permitir uploads públicos durante a apresentação.

Escolha a solução mais simples e estável:

* Caso o Cloudinary já esteja integrado e funcional, mantenha-o configurável por variáveis de ambiente.
* Caso não esteja configurado, utilize produtos e imagens previamente cadastrados para a demonstração.
* Não dependa de armazenamento local temporário em produção.
* Não bloqueie o deploy apenas para permitir que visitantes enviem novas imagens.

## Prioridade 2 — Separar demonstração de erro real

O projeto atualmente utiliza dados mockados e fallbacks em algumas telas.

Crie uma variável explícita no front-end:

`VITE_DEMO_MODE`

Comportamento esperado:

### Quando `VITE_DEMO_MODE=true`

* O projeto pode utilizar dados demonstrativos.
* A interface deve indicar discretamente que está em modo de demonstração.
* Contas, pedidos e produtos simulados podem ser utilizados.
* Funcionalidades externas podem ser simuladas.

### Quando `VITE_DEMO_MODE=false`

* Falhas da API não devem ser substituídas silenciosamente por dados falsos.
* O usuário deve visualizar uma mensagem de erro adequada.
* Pedidos fictícios não devem aparecer quando a API estiver indisponível.
* A aplicação deve utilizar os dados reais do back-end.

Revise principalmente:

`frontend/src/lib/api.ts`
`frontend/src/data/demoOrders.ts`
`frontend/src/pages/Login.tsx`
`frontend/src/pages/MyOrders.tsx`
`frontend/src/pages/ArtisanOrders.tsx`

Não remova todos os mocks obrigatoriamente. Apenas deixe seu uso explícito e controlado.

### Rotas protegidas

Centralize a proteção das rotas, caso isso possa ser feito sem grande refatoração.

Crie componentes equivalentes a:

* `ProtectedRoute`;
* `AdminRoute`;
* `SellerRoute`.

Caso a arquitetura atual já funcione corretamente, faça apenas os ajustes necessários para evitar:

* Exibição temporária de páginas protegidas;
* Acesso sem autenticação;
* Repetição excessiva de verificações;
* Redirecionamentos incorretos.

### Revalidação da sessão

Revise:

`frontend/src/store/auth.ts`

Ao carregar a aplicação:

* Revalide o usuário com `/me` ou endpoint equivalente.
* Atualize os dados da sessão quando forem válidos.
* Limpe a sessão quando o token estiver inválido ou expirado.
* Mostre um carregamento curto durante a validação.

Não altere a autenticação para cookie `httpOnly` nesta etapa caso isso exija uma refatoração ampla.

## Prioridade 4 — Remover funcionalidades visualmente quebradas

Não deixe elementos importantes sem funcionamento.

### Favoritos

Atualmente o back-end possui favoritos, mas o front-end utiliza Zustand/localStorage e o botão do header não leva a uma página funcional.

Escolha a alternativa mais simples e segura:

#### Alternativa preferencial

* Criar uma página simples de favoritos.
* Fazer o botão do header navegar para ela.
* Utilizar o estado local já existente.
* Para usuários autenticados, integrar ao back-end somente se isso puder ser feito sem grande complexidade.

#### Alternativa simplificada

* Remover temporariamente o botão do header.
* Remover chamadas visuais para favoritos.
* Documentar a funcionalidade como melhoria futura.

Não implemente sincronização complexa caso isso aumente muito o escopo.

### Busca

A busca do header atualmente utiliza dados mockados.

Transforme-a em uma busca simples e funcional:

* Pesquisar produtos já carregados no catálogo; ou
* Utilizar um endpoint existente do back-end.

Não é necessário criar autocomplete avançado.

Garanta apenas:

* Campo funcional;
* Resultados coerentes;
* Mensagem quando nenhum produto for encontrado;
* Nenhum resultado fictício quando o modo demonstração estiver desativado.

### Botões e links

Revise o projeto e corrija:

* Botões sem ação;
* Links apontando para páginas inexistentes;
* Imagens quebradas;
* Rotas com erro;
* Páginas vazias;
* Textos provisórios;
* Referências ao projeto antigo.

Quando uma funcionalidade não for necessária para a demonstração, prefira remover o acesso visual em vez de manter algo quebrado.

## Prioridade 5 — Banco e dados de demonstração

Prepare o banco apenas no nível necessário para o deploy.

Garanta que:

* O Prisma utilize `DATABASE_URL`.
* `prisma generate` funcione.
* `prisma migrate deploy` possa ser executado.
* Não seja necessário usar `prisma migrate dev` em produção.
* O schema seja validado.

Crie ou ajuste um seed seguro e idempotente para cadastrar:

* Categorias;
* Planos necessários;
* Produtos demonstrativos aprovados;
* Um usuário administrador demonstrativo, somente quando configurado;
* Um vendedor demonstrativo;
* Um cliente demonstrativo.

Requisitos:

* Não usar dados pessoais reais.
* Não apagar dados existentes.
* Não duplicar registros a cada execução.
* Não colocar senhas reais diretamente no repositório.
* Utilizar variáveis de ambiente para credenciais demo quando necessário.
* Não executar o seed automaticamente sem intenção.

Não é necessário implementar backup automatizado para esta versão de portfólio.

Apenas documente que backup seria necessário em uma aplicação comercial.
